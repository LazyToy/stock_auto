# 🎯 하네스 엔지니어링 템플릿 v2.0

> **최종 업데이트**: 2026-04-04
> **용도**: AI 에이전트 간 作業 위임을 위한 구현 지시 + 검증 체크리스트 표준 형식
> **원칙**: 구현 AI와 검증 AI가 **독립적으로** 이 문서만 보고 작업할 수 있어야 한다.

---

## 📐 구조 설계 원칙

### 왜 하네스 엔지니어링인가?
- AI 모델은 "말(Horse)" — 강력하지만 방향 설정이 필요
- 하네스는 "마구(Harness)" — 제약 조건, 가드레일, 피드백 루프의 집합
- **목표**: AI가 구현 중 표류(drift)하지 않도록 행동 계약(Behavioral Contract)을 정의

### 핵심 원칙
1. **검증 우선(Verification-First)**: 구현보다 "완료 판정 기준"을 먼저 정의
2. **원자적 완결성(Atomic Completeness)**: 각 이슈가 독립적으로 구현/검증 가능
3. **명시적 경계(Explicit Boundaries)**: "해야 할 것"과 "하지 말아야 할 것"을 모두 명시
4. **살아있는 문서(Living Spec)**: 구현 중 발견된 문제로 문서가 업데이트될 수 있음

---

## 📄 전체 문서 구조

```
1. 메타데이터
2. 공통 컨텍스트
3. 이슈 목록 (의존성 맵 포함)
4. 이슈별 상세 (×N)
   4.1 문제 정의
   4.2 구현 지시
   4.3 완료 판정 기준
   4.4 검증 지시
5. 실행 계획
6. 장애 대응
```

## 1. 메타데이터

```markdown
# 🎯 {프로젝트명} — 하네스 엔지니어링 문서

| 항목 | 값 |
|------|-----|
| 문서 버전 | v1.0 |
| 작성일 | YYYY-MM-DD |
| 작성자 | {사람 또는 AI 식별자} |
| 대상 프로젝트 | {프로젝트명} |
| 이슈 수 | N개 |
| 예상 총 소요 시간 | {시간 단위} |
```

> **왜 필요한가**: 여러 하네스 문서가 존재할 때 버전 충돌 방지.
> 이전 문서와 현재 문서 간 이슈 번호 충돌이 없도록 관리.

## 2. 공통 컨텍스트

> 모든 이슈의 구현/검증 프롬프트 앞에 주입되는 프로젝트 정보.
> AI가 "처음 보는 프로젝트"라고 가정하고 필요한 모든 정보를 포함해야 함.

```markdown
## 🔑 공통 컨텍스트

### 역할 지정
너는 {프로젝트명} 프로젝트의 {역할}이야.

### 기술 스택
- {프레임워크 + 버전}
- {빌드 도구}
- {주요 라이브러리}

### 핵심 파일 맵(예시)
| 파일 | 역할 | 현재 줄 수 |
|------|------|-----------|
| `src/App.tsx` | 메인 UI | ~1900줄 |
| `src/lib/core.ts` | 핵심 로직 | ~500줄 |

### 코딩 규칙 (Always/Never)
**Always:**
- 모든 주석/에러 메시지는 {언어}로 작성
- 변경 후 반드시 `npm run lint` + `npm run test` 실행
- 에러 발생 가능 구간에 try-catch 포함

**Never:**
- 기존 API 인터페이스를 breaking change 하지 않음
- 테스트 없이 로직 변경 금지
- console.log를 production 코드에 남기지 않음

### 보고 형식
변경 완료 후 반드시 아래 형식으로 보고:
- ✅ 수정/추가된 파일 목록
- ✅ 변경 내용 (diff 형식)
- ✅ 변경 이유
- ✅ 영향 범위
```

> **이전 버전(harnes.md)과의 차이**: 기존 문서는 컨텍스트를 코드블록 안에 
> 평문으로 넣었으나, Always/Never 리스트와 파일 맵 테이블로 구조화하여 
> AI가 핵심 제약을 놓칠 확률을 줄임.

## 3. 이슈 목록 + 의존성 맵

```markdown
## 이슈 목록

| # | 이슈 | 복잡도 | 핵심 파일 | 선행 이슈 |
|---|------|--------|----------|----------|
| 1 | {제목} | ⬜ 낮음 | `file.ts` | 없음 |
| 2 | {제목} | 🟨 중간 | `file.ts`, `other.ts` | 없음 |
| 3 | {제목} | 🟥 높음 | `file.ts` | #2 |

### 의존성 다이어그램
#1 ──→ #3
#2 ──→ #3
#4 (독립)
```

> **이전 버전과의 차이**: 복잡도 표시와 선행 이슈(의존성) 컬럼 추가.
> AI가 실행 순서를 자체 판단하지 않고, 의존성 그래프를 따르도록 강제.

## 4. 이슈별 상세 — 표준 구조

> 모든 이슈는 아래 4개 섹션을 **반드시** 포함해야 합니다.

```markdown
## 이슈 N: {이슈 제목}

### 4.1 🔍 문제 정의

**현상**: {사용자가 경험하는 문제를 1~2문장으로}

**근본 원인**:
1. {원인 1}: {코드 위치} — {왜 문제인지}
2. {원인 2}: {코드 위치} — {왜 문제인지}

**영향 범위**: {이 문제로 인해 영향받는 기능/파일 목록}

---

### 4.2 📦 구현 지시

**전제 조건**:
- {이 이슈 시작 전에 반드시 참이어야 하는 것들}
- 예: "이슈 #2가 완료되어 있어야 함"
- 예: "`npm run test` 전체 통과 상태"

**수정 대상 파일**: {파일1}, {파일2}

**구현 프롬프트**:
```
[작업 지시]
harnes 문서 "이슈 N: {제목}"을 구현해줘.

[구현 단계]

== 단계 1: {소제목} ==
{파일명} 수정:

- 현재 코드 (L{시작}~L{끝}):
  {기존 코드 요약 또는 발췌}

- 변경할 코드:
  {새 코드}

- 변경 이유:
  {왜 이 변경이 필요한지}

== 단계 2: {소제목} ==
...

[경계 조건 — 하지 말 것]
- {금지 사항 1}
- {금지 사항 2}

[완료 조건]
- {조건 1: 측정 가능한 기준}
- {조건 2: 측정 가능한 기준}
- npm run lint + test 통과
- 수정 파일 목록 + diff 보고
```

---

### 4.3 ✅ 완료 판정 기준 (Acceptance Criteria)

> 구현 AI가 스스로 완료 여부를 판단하는 체크리스트.
> 모든 항목이 ✅일 때만 "구현 완료"로 선언할 수 있음.

| # | 기준 | 검증 방법 |
|---|------|----------|
| AC-1 | {기능적 요구사항} | {어떻게 확인하는지} |
| AC-2 | {비기능적 요구사항} | {어떻게 확인하는지} |
| AC-3 | 기존 기능에 regression 없음 | `npm run test` 전체 통과 |
| AC-4 | 린트 에러 없음 | `npm run lint` 통과 |

---

### 4.4 🔍 검증 지시

> 검증 AI가 구현 결과를 독립적으로 평가하는 프롬프트.
> 구현 AI와 **다른 AI**가 실행한다고 가정하고 작성.

**검증 프롬프트**:
```
[작업 지시]
이슈 N "{제목}" 구현이 완료되었다고 합니다. 검증해주세요.

[1단계: 정적 검증]
1. npm run lint + npm run test 실행
2. 코드 리뷰:
   □ {체크 항목 1}
   □ {체크 항목 2}
   □ {체크 항목 3}

[2단계: 동적 검증 (브라우저/런타임)]
□ {시나리오 1}: {기대 결과}
□ {시나리오 2}: {기대 결과}
□ {에지 케이스}: {기대 결과}

[3단계: 회귀 검증]
□ 기존 기능 A 정상 동작 확인
□ 기존 기능 B 정상 동작 확인

[보고 형식]
전체 결과: ✅ 통과 / ❌ 실패
- 각 항목별 ✅/❌
- 실패 항목: 재현 단계 + 스크린샷/로그
- 수정 제안 (있는 경우)

## 5. 실행 계획

```markdown
## 📋 실행 순서

### 권장 순서
이슈 #1 → 이슈 #2 → 이슈 #4 → 이슈 #3

### 순서 결정 근거
- #1은 독립적이고 간단 → 먼저 처리하여 빠른 성공 확보
- #2는 #3의 전제 조건 → 반드시 선행
- #4는 독립적이지만 #2 완료 후 체감 확인 필요
- #3은 #2 의존 → 마지막

### 병렬 실행 가능 그룹
- 그룹 A (병렬 가능): #1, #4
- 그룹 B (#2 완료 후): #3
```

## 6. 장애 대응

```markdown
## 🚨 장애 대응

### 검증 실패 시
[이전 구현에서 아래 검증 결과가 나왔어. 수정해줘]
{검증 AI의 보고서 붙여넣기}
해당 이슈의 구현 지시를 다시 확인하고 위 버그들을 모두 수정해줘.

### 구현 중 문서와 충돌 발견 시
1. 구현을 중단
2. 충돌 내용을 보고 (어떤 지시가 현재 코드와 맞지 않는지)
3. 문서 수정 후 재개

### 롤백이 필요한 경우
1. `git stash` 또는 `git checkout -- .` 으로 변경 사항 복원
2. 실패 원인 분석 보고
3. 문서의 구현 지시를 수정한 후 재시도
```
